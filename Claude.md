# ТЗ: пул предсказаний (pari-mutuel) на TON / Tolk

> **Задача для Claude Code:** создать проект смарт-контракта по этому ТЗ с нуля — контракт на Tolk, нативные тесты и скрипты на Tolk, обёртки генерируются Acton'ом (стек **Acton**).

## 1. Что строим

Бинарный пул предсказаний по pari-mutuel схеме: участники ставят TON на исход (YES/NO) до дедлайна, владелец фиксирует реальный исход, победители забирают банк пропорционально (pro-rata) своим ставкам через pull (claim). Это учебный проект для освоения архитектуры TON: словари (hashmap), логическое время, обработка входящих сообщений по `op`, get-методы, асинхронные выплаты и обработка bounce.

## 3. Жизненный цикл

1. **Деплой.** Владелец задаёт параметры (см. §4 «Параметры деплоя»): `bettingDeadline`, `resolutionDeadline`, `minBet`, `feeBps` — и закладывает стартовый баланс (**сид**), покрывающий ренту за весь период и газ на операции (см. §8). Валидация параметров — см. §8.
2. **Приём ставок** (`now < bettingDeadline`): любой шлёт TON с выбором YES/NO.
3. **Закрытие.** После `bettingDeadline` ставки не принимаются.
4. **Резолв** (`bettingDeadline <= now < resolutionDeadline`): владелец сообщает выигравший исход.
5. **Выплаты.** Победители через `claim` забирают pro-rata долю.
6. **Аварийная отмена.** Если владелец не резолвнул до `resolutionDeadline`, любой может перевести пул в `CANCELLED`, после чего все возвращают свои ставки.

## 4. Состояние (storage)

- `ownerAddress`
- `bettingDeadline: uint` (unix)
- `resolutionDeadline: uint` (unix)
- `minBet: coins` — минимальная ставка (задаётся при деплое; в этом проекте **0.5 TON**)
- `status: uint` — enum: `OPEN | RESOLVED | CANCELLED`
- `winningOutcome: uint` — `0` = не задан; иначе `YES = 1`, `NO = 2`
- `totalPot: coins`
- `totalYes: coins`, `totalNo: coins`
- `feeBps: uint` (базисные пункты; в этом проекте **500 = 5%**)
- `bets: dict` — ключ: 256-битный хэш адреса участника (basechain, workchain 0); значение: `(outcome: uint, amount: coins, claimed: bool)`
- счётчики для безопасного вывода: число адресов с правом на выплату/возврат и число сделанных клеймов (см. §7).

**Параметры деплоя:** `ownerAddress` (= деплойер), `bettingDeadline`, `resolutionDeadline`, `minBet` (0.5 TON), `feeBps` (500), плюс стартовый баланс (**сид**) на ренту/газ (см. §8). Кодировка исходов единая для ставок и `winningOutcome`: `YES = 1`, `NO = 2`.

## 5. Сообщения (ops)

- **`bet`** (параметр: исход; к сообщению приложены монеты). Принять только если `status == OPEN`, `now < bettingDeadline`, приложенная сумма `>= minBet`, исход валиден (`YES`/`NO`). **Ставка = весь приложенный TON (`in.value`)**: газ из ставки не вычитается — он покрывается комиссией (см. §7). Записать/накопить ставку, обновить `totalYes/totalNo` и `totalPot`. Повторная ставка того же адреса на **тот же** исход — суммируется; на **другой** исход — отклонять.
- **`resolve`** (только владелец; параметр: исход). Только `status == OPEN`, `bettingDeadline <= now < resolutionDeadline`, исход валиден. Установить `winningOutcome`, `status = RESOLVED`. Повторный резолв отклонять. **Если на выигравшую сторону никто не ставил (`winSide == 0`) — не переводить в `RESOLVED`, а сразу `status = CANCELLED`** (иначе банк застрянет; см. §7).
- **`claim`** (отправитель забирает выплату/возврат). Только при `status ∈ {RESOLVED, CANCELLED}`. У отправителя должна быть ставка и `claimed == false`. Если `RESOLVED` — исход ставки должен совпасть с `winningOutcome`. Пометить `claimed = true` **до** отправки, затем отправить выплату. Победитель получает **полную** pro-rata сумму; газ на исходящую выплату идёт из баланса контракта (комиссии), а не вычитается из выплаты.
- **`cancel`** (любой). Только `status == OPEN` и `now >= resolutionDeadline`. Перевести в `CANCELLED`.
- **`withdrawFee`** (только владелец). Вывести оунеру причитающийся остаток после того, как все выплаты/возвраты сделаны (или по истечении грейс-периода — см. §7).
  - **RESOLVED:** комиссия (`fee`) + остаток от деления (dust) + неизрасходованный газ + остаток сида.
  - **CANCELLED:** комиссии нет — возвращается только **остаток сида** (баланс сверх ещё не востребованных возвратов).
  - В обоих случаях вывод не должен опускать баланс ниже суммы невостребованных выплат/возвратов + резерв на хранение (см. §7, §8).

Отклонение любого `op` — это `throw`: входящее сообщение bounce-нется и приложенные монеты вернутся отправителю (минус комиссии сети). Поэтому `bet` слать как bounceable.

## 6. Get-методы

> Имена в camelCase — под стиль геттеров Acton-скаффолда (`currentCounter`, `owner`).

- `getStatus()` → `status, winningOutcome, bettingDeadline, resolutionDeadline`
- `getPool()` → `totalPot, totalYes, totalNo, feeBps`
- `getBet(address)` → `outcome, amount, claimed`
- `getPayout(address)` → сколько адрес может забрать сейчас (`0`, если не положено)

## 7. Логика выплат (pro-rata, целочисленно)

- Комиссия: `fee = totalPot * feeBps / 10000` (при `feeBps = 500` это 5% банка); `distributable = totalPot - fee`.
- Сторона-победитель: `winSide = (winningOutcome == YES) ? totalYes : totalNo`.
- Выплата участнику `W`: `payout = W.amount * distributable / winSide`. **Умножать до деления** ради точности. Победитель получает эту сумму целиком. (В коде формула раскрыта в одно выражение `amount * totalPot * (10000 − feeBps) / (winSide * 10000)` — все умножения до единственного деления; алгебраически идентично.)
- **Газ — из комиссии.** Все операционные издержки (форвард-фи на исходящих `claim`/`withdrawFee`, рента за хранение) покрываются балансом контракта (сид + комиссия); выплаты победителям на газ не уменьшаются. `feeBps` должен покрывать совокупный газ — при 5% от ставок `>= 0.5 TON` это выполняется с запасом.
- **Остаток оунеру.** Остаток от деления (dust), неизрасходованный газ и остаток сида копятся в контракте. После завершения всех клеймов/возвратов владелец забирает их через `withdrawFee`: при **RESOLVED** — `fee` + dust + неизрасходованный газ + остаток сида; при **CANCELLED** — комиссии нет, только **остаток сида** (`сид − издержки`).
- **Защита выплат при выводе (инвариант).** `withdrawFee` не должен уводить баланс ниже `(невостребованные выплаты/возвраты) + резерв на хранение`, иначе не хватит на ещё не сделанные клеймы. Рекомендуемый механизм без итерации по словарю: вести счётчики «число адресов с правом на выплату/возврат» (победители при RESOLVED — известно на `resolve` из числа ставок на сторону; все вкладчики при CANCELLED) и «число сделанных клеймов»; разрешать полный вывод, когда они равны. Чтобы средства не залипли, если кто-то не клеймит, дополнительно разрешить вывод после грейс-периода за `resolutionDeadline` (поздний неклейм — в пользу оунера; задокументировать).
- Если `winSide == 0` — пул переводится в `CANCELLED` ещё на `resolve` (см. §5): все возвращают ставки.
- При `CANCELLED`: `payout = W.amount` (полный возврат, без комиссии); остаток сида после возвратов забирает оунер (см. выше).

## 8. Безопасность и крайние случаи

- **Валидация при деплое:** `now < bettingDeadline < resolutionDeadline`; `0 <= feeBps <= 10000`; `minBet > 0`; приложенный сид `>= minSeed`. Иначе инициализация/деплой отклоняется.
- **Стартовый сид.** Оунер обязан приложить при деплое начальный баланс, покрывающий резерв на хранение + газ на операции (резолв, вывод) + ренту за весь период. Ориентир `minSeed` — не меньше ~`0.1 TON` (резерв `0.05` + буфер); конкретное значение — на усмотрение реализации.
- Ставки после `bettingDeadline` — отклонять; нулевые/мелкие (`< minBet`) — отклонять.
- Резолвить может только владелец, только в окне `[bettingDeadline, resolutionDeadline)`, только один раз.
- Клейм — только после резолва/отмены, только победителю (или всем при отмене), только один раз на адрес.
- **Асинхронность:** помечать `claimed` до отправки; если исходящая выплата **bounce-нется**, откатывать `claimed` в обработчике bounce (`onBouncedMessage`), чтобы средства не потерялись. Плательщика определять по `senderAddress` bounce-сообщения.
- **Резерв на хранение:** контракт всегда держит минимум на ренту (ориентир — `0.05 TON`); если `op` не оставляет резерв — отклонять.
- Целочисленная арифметика без переполнений; остаток (dust) — по правилу из §7.
- Кодировка исходов: `YES = 1`, `NO = 2`; `0` невалиден для ставки/резолва.

## 9. Коды ошибок (exit codes)

Именованные коды (в стиле `Errors.X` Acton-скаффолда); на них опираются тесты отклонений (§10). Конкретные числовые значения — на усмотрение реализации, важна стабильность имён.

- `NotOwner` — операция требует владельца.
- `InvalidMessage` — неизвестный `op` / некорректное тело.
- `InvalidOutcome` — исход не `YES`/`NO`.
- `BettingClosed` — ставка после `bettingDeadline` или `status != OPEN`.
- `BetTooSmall` — приложенная сумма `< minBet`.
- `OutcomeConflict` — повторная ставка на другой исход.
- `ResolveTooEarly` / `ResolveTooLate` — резолв вне окна `[bettingDeadline, resolutionDeadline)`.
- `AlreadyResolved` — повторный резолв / `status != OPEN`.
- `NotResolved` — клейм/вывод до резолва или отмены.
- `CancelTooEarly` — `cancel` до `resolutionDeadline`.
- `NoBet` — у отправителя нет ставки.
- `AlreadyClaimed` — повторный клейм.
- `NotAWinner` — клейм проигравшего при `RESOLVED`.
- `WithdrawTooEarly` — вывод оунером до того, как выплаты/возвраты сделаны (и грейс не наступил).
- `SeedTooSmall` — при деплое приложен сид меньше `minSeed`.
- `InsufficientReserve` — не остаётся резерв на хранение/выплаты.

## 10. Тесты (Acton, `*.test.tolk`)

- **Happy path:** 2 ставки на YES (разные суммы) + 1 на NO, закрытие, резолв YES, оба YES-победителя забирают верную pro-rata долю (за вычетом 5% комиссии), NO — ничего, повторный клейм отклонён (`AlreadyClaimed`). После клеймов оунер забирает `fee + dust + остаток сида`.
- **Нет победителей:** все на NO, резолв YES → `winSide == 0` → авто-`CANCELLED` → возврат всем; после возвратов оунер забирает остаток сида (комиссии нет).
- **Не резолвнули:** после `resolutionDeadline` любой делает `cancel` → все возвращают ставки; затем оунер забирает остаток сида.
- **Отклонения:** поздняя ставка (`BettingClosed`); ставка `< minBet` (`BetTooSmall`); резолв не-владельцем (`NotOwner`); резолв до дедлайна (`ResolveTooEarly`) / после `resolutionDeadline` (`ResolveTooLate`); конфликтующая повторная ставка на другой исход (`OutcomeConflict`); клейм до резолва (`NotResolved`); вывод оунером до завершения клеймов (`WithdrawTooEarly`); деплой с малым сидом (`SeedTooSmall`).
- **Комиссия (5%):** владелец получает комиссию + остаток, победители — банк за вычетом комиссии; суммы сходятся (банк = выплаты + fee + dust, газ — из комиссии/сида).

## 13. Известное ограничение доверия

Резолв исхода делает владелец — это централизованная точка доверия (оракул-проблема). Защитный клапан — `resolutionDeadline` + `cancel`: если владелец не резолвнул вовремя, ставки возвращаются и участники не застревают. Для продакшена эту точку нужно заменить оракулом или механизмом споров.
